LWA モバイル SDK を使用 (DRS)

概要

Dash Replenishment Service（DRS、Dash 補充サービス）にアクセスするには、DRS を有効化した貴殿/貴社の製品が「Amazon にログイン」アクセストークンを取得する必要があり、それは個々のリクエストと一緒に送信されます。 本書では Android または iOS 用の 「Amazon にログイン」 SDK を使用してアクセストークンを取得する方法を説明します。

前提条件

LWA SDK を貴殿/貴社のコンパニオンアプリに統合する前に、貴殿/貴社の製品はコードベリファイアを生成しコードチャレンジを作成できなければなりません。 コードチャレンジ方式によるこれらの値は、トークンが交換される前に貴殿/貴社の製品からのリクエストを検証するため LWA により使用されます。 LWA による対称型の所有証明（proof of possession）の実装は OAuth 公共クライアントによるコード交換用証明鍵に基きます。

  • コードベリファイア

コードベリファイアとは貴殿/貴社の製品が生成する暗号学的にランダムなストリングで、ハッシュ値（SHA256）として貴殿/貴社のコンパニオンアプリに引き継がれます。 ストリングは43～128の文字数を含み、URL からの文字およびファイル名として安全な英数字（[A-Z]、[a-z]、[0-9]、「-」、「_」、「.」、「~」）から構成されています。

コードベリファイアは機密データであり、決して貴殿/貴社の製品から転送してはなりません。

  • コードチャレンジ

貴殿/貴社のクライアント/製品はコードベリファイアの以下の変換のひとつを用いてコードベリファイア由来のコードチャレンジを作成することになります。変換にはSHA-256 が推奨されます。

  • 平文 コードチャレンジ＝コードベリファイア

    Note: 平文（plain）は S256 変換を使用できない既存のシステムおよび制約の多い環境との互換性のためです。

  • S256 貴殿/貴社のコードベリファイアの SHA256 ハッシュ値の Base64url コード化。 Base64url コード化したストリングは以下の文字を含めません。 「＝」、「＋」、または「/」。

Base64url コード化の実装に関する詳細は OAuth 公共クライアントによるコード交換用証明鍵、付録 Aを参照してください。

  • コードチャレンジ方式

コードチャレンジを派生させるのに使用される方式。 LWA は平文および SHA-256 を受け付けます。

Amazon にログイン SDK の使用

次の項では DRSで Android または iOS 用 Amazon にログイン SDK を使用するための手引を提供します。

以下のタブを使用し貴殿/貴社のモバイル・オペレーティングシステムを選択してください。

以下の指示に従い Android 用 Amazon にログイン SDK を使用し、Amazon にログイン認可コードを貴殿/貴社の製品に渡してください。貴殿/貴社の製品はそれを使用して DRS へのコールに必要なリフレッシュおよびアクセストークンを取得することができます。

  • Android 用 Amazon にログインのスタート に移動し、ステップの1～4を実行してください。
  • 次の各ステップを完了し、DRS へのコールに使用されるアクセスおよびリフレッシュトークンを取得します。
  1. Amazon にログイン API をソースファイルにインポートしてください。 Amazon にログインAPI をインポートするには、ソースファイルに次のインポート用コマンド文を追加します。

  2. AmazonAuthorizationManagerの初期化

    AmazonAuthorizationManager 変数を宣言し、クラスの新規インスタンスを作成する必要があります。 新規のインスタンスの作成に必要なのは、現在のアプリケーション・コンテキストと空のバンドルだけです。 AmazonAuthorizationManager をイニシャライズするのに最良の場所は貴殿/貴社のアクティビティの onCreate 方式内です。 一例

  3. AuthorizeListener を作成。

    AuthorizeListener は AuthorizationListener のインターフェースを実装し、認可コールの結果を処理します。 これには onSuccess、onError、 onCancel の3方式があります。 各方式がバンドルまたは AuthError オブジェクトを受け取ります。

  4. AmazonAuthorizationManager.authorize コール。

    貴殿/貴社の「Amazon にログイン」ボタン用 onClick ハンドラー内で authorize をコールし、ユーザーにログインし貴殿/貴社のアプリケーションを認可するよう促します。 この方式は次の方法のひとつでお客様を認可する役割を担います。

    • システムブラウザに切り替わり、お客様にサインインしリクエストされた情報に同意してもらう
    • 安全なコンテキストでWebViewに切り替わり、お客様にサインインしリクエストされた情報に同意してもらう

    上記2点目 の「安全なコンテキスト」は現在 Android 端末上の Amazon ショッピングアプリとして実現しています。 Fire OS（Kindle タブレットおよび Fire TVなど）が稼働している Amazon デバイスでは、デバイス上に Amazon ショッピングアプリがない場合でも常にこのオプションを使用しています。 このためお客様がすでに Amazon ショッピングアプリにサインインしている場合は、この API はサインインページを省略しますから、お客様にとってはシングルサインオン体験となります。

    貴殿/貴社のアプリケーションが認可されると、スコープとして知られるひとつまたは複数のデータセットに認可されます。 一つ目のパラメータは貴殿/貴社が Amazon にログインからリクエストしているユーザーデータを含むスコープの配列です。 ユーザーが初めて貴殿/貴社のアプリにログインすると、貴殿/貴社がリクエストしているデータの一覧が表示され、承認を求められます。 DRS で使用する Amazon にログインは現在 dash:replenish スコープをサポートしています。

    authorizeをコールする最良の方法は非同期による方法で、この場合 UI スレッドをブロックしたりご自身のワーカースレッドを作成する必要がありません。 authorize に非同期でコールするためには、AuthorizationListener インターフェースをサポートするオブジェクトを最後のパラメータに指定します。

    上記のコードに次のオプションを追加します。

    • scope_data はデバイスモデル、デバイス識別子、およびデバイスがテストデバイスかどうかを含む追加のデータです。

    パラメータ オプション

    クライアントがレスポンスに認可コードを取得したい場合は「真」でなければならないブール値。

    認可リクエストを呼び出すのに貴殿/貴社のデバイスが生成する SPOP コードチャレンジ。 このパラメータは AuthzConstants.BUNDLE_KEY.GET_AUTH_CODE.val が真に設定されている場合は常に必要です。そうでなければSDK はエラーを返します。

    以前のステップでコードチャレンジの生成に使用した SPOP コードチャレンジ方式。 現在 LWA は2つのコードチャレンジ方式をサポートしています。 「平文」および「S256」（SHA-256）。 ただし、コードチャレンジ方式の指定がなければ、LWA は規定値として平文として扱います。

Note: 登録はWebView内で取り扱います。 LWA 認可 API が起動されると、お客様はこのWebViewにリダイレクトされます。 お客様は登録の際にデバイスとスロットを構成（および ASIN の選択）します。 選択が完了した時点で貴殿/貴社のアプリまたはクラウドサービスは DRS API を使用できるようになります。

  1. 貴殿/貴社の AuthorizeListener に onSuccess を実装してください。

    onSuccess で AmazonAuthorizationManager.getToken をコールしアクセストークンを取得します。 getToken は authorize と同様、非同期リスナーインターフェースを使用します。 getToken ではそのインターフェースは APIListener であり、AuthorizationListener ではありません。

  2. AuthorizeListener に onError を実装してください。

  3. AmazonAuthorizationManager の貴殿/貴社のインスタンスから getRedirectUri() および getClientId() をコールし、保存したのち貴殿/貴社の製品に送信します。

    Note: client_id はアクセスおよびリフレッシュトークンの取得のために使用されます。 このクライアント識別子は貴殿/貴社のセキュリティプロフィールページから得られる識別子とは異なります。
  4. SSL を使用してリフレッシュトークン、クライアント ID、およびリダイレクト URI を安全に貴殿/貴社の製品に送信してください。

  5. 製品が認可コード、クライアント ID、およびリダイレクト URI を受領した後、Amazon にログインにコールし、認可コードをリフレッシュおよびアクセストークンと交換する必要があります。 製品はコール時に POST リクエストを https://api.amazon.com/auth/O2/token に送信し、次のパラメータを指定する必要があります。

    • grant_type: authorization_code
    • code: Android アプリから受信した認可コード。
    • redirect_uri: Android アプリから受信したリダイレクト URI。
    • client_id: Android アプリから受信したクライアント ID。
    • code_verifier: 初期に製品が生成したコードベリファイア。

    レスポンスは次の値を含みます。

    • access_token: アクセストークン。
    • refresh_token: リフレッシュトークン。
    • token_type: ベアラー
    • expires_in: アクセストークンが引き続き有効である秒数。

  6. アクセストークンが時間切れになったとき、またはなりつつあるとき、リフレッシュトークンを新規のアクセストークンと交換できます。 交換を行うには、https://api.amazon.com/auth/O2/token に POST し、次のパラメータを送信します。

    • grant_type: refresh_token
    • refresh_token: リフレッシュトークン。
    • client_id: Android アプリから受信したクライアント ID。

    レスポンスは次の値を含みます。

    • access_token: アクセストークン。
    • refresh_token: リフレッシュトークン。
    • token_type: ベアラー\
    • expires_in: アクセストークンが引き続き有効である秒数。

LWA が正常に設定されたことを確かめてください

貴殿/貴社のアプリケーションがお客様を LWA へのログインに誘導できるようになり、かつお客様が LWA を使用して自身のデバイスを登録でき、貴殿/貴社のアプリケーションがリフレッシュトークンおよびアクセストークンを受信したときにLWA リクエストは完了です。 LWA の統合が正常に完了すると、以下の情報が表示されます。

  • client_id
  • client_secret
  • access_code
  • access_token
  • refresh_token
